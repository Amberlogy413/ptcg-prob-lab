# 09 沙盤推演 — 一盤 PTCG 的微觀步驟與真實需求

> 本文以「一位玩家由賽前構築到賽後覆盤」的完整旅程為沙盤,逐步推演每個微觀步驟、
> 每個決策點玩家心中真正想要的數字,並對照本產品(2026-06,v1.1.0)的實際覆蓋。
> 覆蓋判定以原始碼為準(已逐項核對 `src/`),非以願望為準。
>
> 標記:**完整** = 有直接入口且數學正確;**部分** = 數字可間接得到或覆蓋一半;
> **無** = 產品內完全沒有;**刻意不做** = 落在不做清單(勝率預測、對手模型、
> 比賽中即時輔助),屬有意的誠實邊界。

---

## §1 全盤微觀流程圖(六段)

### 第一段:賽前構築與檢查(由揀套路到上桌)

1. 揀定套路(netdeck 或自起骨架)→ 候選牌表二揀一
2. 牌表搬入工具:匯入精靈(Live 格式/「4 卡名」/「卡名 x4」)、視覺組牌、範本骨架
3. 標記基礎寶可夢(補完標籤/全域記憶/社群名單/逐行手動)— Mulligan(重抽)數學的必要輸入
4. 核對總張數 = 60(計數環)
5. 構築規則自查:同名卡 ≤4(基本能量豁免)、ACE SPEC ≤1、輝光 ≤1
6. 賽制合法性自查:逐張確認標準賽制合法、無退環境卡
7. 審視基礎寶可夢總數 B 與重抽概率;微調 B 的邊際(±1 隻細基礎)
8. 定進化線投數(4-3-3 / 4-4 / 2-2)、估算到第 T 回合齊件率
9. 驗證起手 combo 成形率(含重抽條件化)
10. 定能量張數(斷能風險 vs 牌位)、檢索/抽濾引擎投數(等效摺疊)
11. 揀 tech 卡(被放入獎賞卡風險、到手回合)、關鍵卡 3 投或 4 投、分配自由格
12. 定義理想手/可用手,檢視死手率與死手歸因
13. 跑全套數學體檢報告,與改版前做 A/B 對照;試抽桌/goldfish 試手感
14. (賽期跨輪替時)跑輪替預覽
15. 填寫賽事登錄牌表並列印簽名 → 實卡執齊 60 張逐張核對 → 入牌套 → 賽前洗牌,上桌

### 第二段:開局程序(洗牌到第一回合限制)

1. 雙方亮牌組、充分洗牌(概率模型假設:完美均勻洗牌,02 §0);對手可切牌
2. 擲硬幣定勝方,勝方選先攻或後攻
3. 各自抽起手 7 張,點算基礎寶可夢張數 k
4. k = 0 → 宣告 Mulligan(重抽):亮全手、洗回、重抽,直到 ≥1 張基礎寶可夢;記錄重抽次數
5. 選 1 張基礎寶可夢面朝下放戰鬥區(強制);可選放最多 5 張到備戰區
6. 牌庫頂取 6 張面朝下作獎賞卡(此刻牌庫剩 47 張)
7. 雙方設置完成後同時翻面;重抽補償結算:對手每重抽 1 次,你「可以」多抽 1 張
   (時機在設置完成之後 — 補償抽救不了擺場,也不影響已放置的獎賞卡)
8. 先攻玩家回合開始,抽 1 張(現行規則先攻第一回合照抽;「先攻不抽」是歷史規則,
   工具以開關保留、預設關)
9. 先攻第一回合限制:不能攻擊、不能使用支援者卡;後攻無此限制 — 先後攻選擇的核心代價

### 第三段:前期回合 T1–T3(鋪場與成形)

1. 確認先攻/後攻與第一回合限制,排定回合計劃
2. 每回合開始抽 1 張(對手重抽的補償張數一併計入「已見」)
3. 盤點手牌:基礎寶可夢、能量、檢索物品、支援者各幾張,排定行動次序
4. 決定備戰區鋪幾多隻基礎寶可夢(防斷後備 vs 防點名集火)
5. 使用球類/檢索物品:決定次序與目標;翻閱牌庫時順手清點關鍵卡剩餘張數
   (反推被獎賞卡收走的張數)
6. 支援者選擇:抽卡型(棄手抽 7)vs 檢索型(精準取 1)
7. 貼本回合唯一一粒能量(主攻線 vs 後備線);需要時計撤退費與棄能代價
8. T2:檢查 1 進化體是否到手,決定即刻進化或等線齊;第二粒能量分配
9. T3:combo 成形總檢(神奇糖果 + 2 進化、招式能量到位);糖果時點判斷
10. 連續幾回合未見關鍵卡:判斷是「黑」、「被獎賞」還是構築問題

### 第四段:中期回合(獎賞交換與資源戰)

1. 對手 KO 我方寶可夢取走獎賞卡 → 即時更新雙方獎賞比數與對手手牌張數
2. 回合開始前盤點公開資訊:牌庫剩餘、棄牌區、已見張數 — 更新「未見集合」心帳
3. 抽回合卡(top deck 時刻),即時記入「已見」
4. 揀新前排:攻擊手接力 vs 犧牲品拖一回合 — 框定獎賞交換節奏
5. 數 prize race:雙方剩餘獎賞 ÷ 每次 KO 取獎賞數 → 各需幾多次 KO、邊個先到 0
6. 計傷害數學:基礎傷 + 修正逐項加、弱點 ×2 / 抗性 −30,得出「差幾多先夠 KO」
7. 差傷害時翻查資源:牌庫剩幾多張 buff 未見、今回合仲有幾多次抽/檢索機會
8. 執行檢索鏈(每次翻牌庫都是新「已見」,獎賞卡後驗即時更新);
   判斷關鍵卡(如 Boss)仍在牌庫或已被獎賞,決定是否燒檢索資源
9. 能量加速判斷:今回合起技夠不夠、下回合斷流風險
10. Gust 決策:拉後排收 2 張獎賞 vs 留作終盤一槍;攻擊宣言前最後覆核 KO 線
11. KO → 取獎賞卡(6 張面朝下揀一張 — 數學上揀邊張完全等價)→ 更新已見帳
12. 回合收尾防 Iono/N:手牌肥就考慮先使凸資源;對手 Iono 後即場評估新手接唔接到線
13. (覆盤時)重行成個回合,逐個決定攞返背後個數驗證

### 第五段:終盤(outs、計時與落注)

1. 盤點未見集合:u = 牌庫剩餘 + 我未翻開的獎賞卡;雙方獎賞比數
2. 列出收尾路線(一擊 2 / 兩擊 1+1 / 迫對手抽死),對每條路線數 outs
   (總投數 − 已見 = 未見 u_x;提醒自己部分解可能瞓在獎賞卡)
3. 回合抽卡 = top deck 時刻:評估「呢一抽中解」的真實概率
4. 決定使不使抽牌資源(博士研究員抽 7 / Iono 抽 = 自己獎賞數)
5. 檢查關鍵 1 投卡(ACE SPEC / 唯一 Boss)被困獎賞卡的後驗,決定路線是否避開
6. 檢索與抽牌的先後次序(先抽薄牌庫 vs 先檢索)
7. 攻擊取獎賞,翻開後即時更新(已見 +1、u−1、獎賞剩 p−1),重算所有後驗
8. 預判對手 comeback(Iono 將我手牌洗回牌庫底、counter catcher 等落後限定卡)
9. 計自己 deck-out 風險:牌庫剩 X 張 = 最多 X 個自然抽;收尾需時 k 回合 ≤ X?
10. (stall 路線)對手牌庫剩 Y 張 → 佢最多 Y 個回合;我方防守卡要接力 Y 回合
11. 時間到 +3 回合程序:確認自己在 turn 幾、剩餘抽卡次數內取唔取得切最後獎賞
12. 終局落注:贏法只剩 top deck 時,心中要有個數 — 係 30% 定 5% 的輸家
13. 賽後重建終盤狀態,逐個決定驗證

### 第六段:賽後覆盤與訓練閉環

1. 單局結束即時留下敗因第一印象(起手?斷流?王牌被獎賞?打錯?)
2. 局間 60–90 秒快速自我歸因(賽中不可開工具,只靠賽前內化的概率直覺)
3. 賽事中草記每局:對手牌型、勝負、重抽次數、關鍵卡首見回合、被獎賞的卡
4. 回家逐局重組敗因:死手 / 斷流 / 相性 / 操作
5. 對每個「我好黑」主張做誠實檢驗:精確概率 vs 實際發生次數,計「有幾出奇」
6. 判詞分流:運氣異常 → 唔使改牌;構築問題 → 入構築循環;操作問題 → 入練習循環
7. 構築線:體檢報告 → 死手歸因排行榜 → 0–4 投敏感度掃描 → fork + A/B 比較 →
   落實改牌(並保留舊版)→ 試抽桌 ×100 驗證 → goldfish 試新劇本
8. 練習線:試抽一手轉訓練題 → 心算 → 揭曉精確值 → 誤差入紀錄;
   謬誤博物館重溫系統性錯估;實體練習局用獎賞卡追蹤器練後驗更新紀律
9. 賽前夜:重跑 60 秒體檢、出數字圖卡/列印牌表、記熟關鍵數字 — 閉環完成

## §3 缺口分析(現有覆蓋 = 無,按結構分組)

### 3.1 對局中條件化引擎缺位(最大結構性缺口)

橫跨 T1 到終盤,玩家九成問題都是「**由而家狀態出發**」的條件超幾何:未見 u 張、
目標未見 u_x 張、再見 d 張、獎賞剩 p 張。數學上**零新公式** — 02 §5.5 可交換性
已證「給定已見,未見集 U 上均勻」,核心 `hypergeom.ts` / `prizes.ts` 本身全參數化;
缺的只是規格小節 + UI。受影響決策點:

- **#36 / #58** 下一抽 = u_x/u(top deck、終局落注 — 三格式+收據簽名最應出現之處)
- **#37 / #41 / #50** 未來 d 抽窗口(後備基礎、能量斷流中局版、buff 補件)
- **#40 / #53** 棄手抽 7 / Iono 重建(自由 N′/H′ 的 Q2 聯合;含「手牌洗回牌庫」情境)
- **#33** 補償抽 7+m 張手牌的組合查詢(Q2 UI 寫死 H=7)
- **#57 / #61** 由當前狀態起的收尾路線 / stall 接力(嵌套超幾何換起點)
- **#59** 博士抽 7 / Iono 抽 p 張至少中 1 解 + 對 top deck 的 delta
- **#38** 檢索摺疊的「到 T 回合窗口」版
- **#62** +3 回合 2 抽內見解(賽前訓練題形態)

附帶的系統性謬誤教學位:「**Y/X 謬誤**」— 用牌庫張數做分母計 outs、忽略解可能
被獎賞;正解 u_x/u 一步可推,完全符合 §9 謬誤清單哲學,目前無任何展示。

### 3.2 追蹤器正確性與證據模型(現有功能的誠實債)

- **#52(正確性缺口)** `computeTrackerRows`(src/state/q5.ts)經 `PRIZE_COUNT=6`
  常數(src/constants.ts)假設獎賞永遠 6 張,無「已攞獎賞 k 張」輸入 — 一攞第一張
  獎賞卡後驗就錯,而追蹤器最有價值的時刻正正是中後盤。修正是同族超幾何(6−k),
  但必須走 Python 黃金先行的雙實作管線。
- **#39** 證據模型太窄:只接受「已見 +1」;全牌庫檢索後的「牌庫實數直入 /
  確定被獎賞(概率=1)」表達不到;§5.5 明文要求的「看牌庫頂後洗回 → 資訊失效」
  亦無 UI 承載(seen/unseen 二分缺第三類「確定在牌庫但未抽」狀態,#53 同源)。
- **#48(部分)** 分布已算齊,只欠「至少 1 張仍在牌庫」一欄。

### 3.3 構築合法性零驗證(賽前段最大誠實缺口)

- **#6 / #7** 同名卡 ≤4(基本能量豁免)、ACE SPEC ≤1、輝光 ≤1 全程無檢查 —
  玩家可帶住違規牌組一路綠燈行到登錄;計數環只驗總數 60。
- **#8(部分)** 賽制資料每張卡都有,缺一鍵「列出全副牌非標準卡」審計入口。
- **#22(部分)** 登錄牌表欠系列代碼/卡號欄 — 資料在手(補完標籤已有 set+number),
  `deckSheet.ts` 只渲染張數+卡名。

### 3.4 比數與節奏(確定性算術,零概率新數學)

- **#47** prize race 獎賞交換競賽:剩餘獎賞 ÷ 每槍取數(1/2/3)、犧牲品如何改變
  雙方所需 KO 次數 — 中期決策的主框架,完全合規(不需知對手牌組),零覆蓋。
- **#60** deck-out 風險 / 回合競賽計數(我剩 X 抽 vs 路線需時 k vs 對手剩 Y 抽)。
- **#49(部分)** KO 算術(傷害 ×2 弱點 / −30 抗性 + 修正 vs HP)— 目錄數據齊全,
  缺一條加數的計算層;要劃清界線不滑向對局模擬。
- **#62** 時間到 +3 回合 / sudden death 程序 — 規則事實零覆蓋;受「比賽中即時輔助
  不做」鐵律約束,正確形態是訓練題型與覆盤重建。

### 3.5 覆盤數據鏈斷層(07 §C1 前半)

- **#64** 對局紀錄完全未實現 — app 無處落帳實戰數據,「實測 vs 理論」判詞(#66)
  因此出不到;LuckMeter 靠玩家憑記憶報數,記憶偏差污染誠實照鏡的前提。
- **#65** 運氣儀表事件空間太窄:只有 (1−p)^n 連黑、事件寫死兩款;「n 局中 ≥k 次」
  精確 BigInt 二項尾是順手延伸,需先補 02 §10 規格小節。

### 3.6 訓練閉環半成品

### Phase 11 — 賽事統計(覆盤數據鏈)

主題:把「實測 vs 理論」的最後一里駁通。

1. **#5 運氣儀表 v2(二項尾 + 任意事件)**— 先補 02 §10 規格小節,雙實作管線
2. **#6 對局紀錄簿 + 判詞**— 依賴 #5;判詞與改牌理由可落地、可匯出

   改動副作用一眼睇晒

### Phase 12 — 題庫與評分遊戲(訓練閉環補完)

   「補償抽該唔該抽盡」— 全部賽前訓練形態,避開「比賽中即時輔助」鐵律
4. 謬誤館收新條目(Y/X 謬誤、揀獎賞卡等價)與題庫互相引用

### Phase 13 — 對局內助手(練習與覆盤用,常駐合規提醒)

賽事合規提醒,定位明確為**練習局與覆盤重演**,非比賽中即時輔助。
